# Stellarator β Climb — Specialised Notes for HAO (Np=2, 3D Field)

> **Scope:** This document covers why β climbing in a stellarator (specifically the
> HAO device with field-period number Np=2) is harder than in a tokamak, the
> limitations of the current pyna implementation, recommended numerical settings,
> and the interface with Optuna optimisation loops.

---

## 1. Why β Climbing is Harder in a Stellarator

### 1.1 No simple Grad-Shafranov equation

In an axisymmetric tokamak the force-balance condition reduces to the
scalar **Grad-Shafranov equation**:

```
Δ*ψ = −μ₀ R² p'(ψ) − F F'(ψ)
```

which lives on a 2D (R, Z) domain and can be solved efficiently with FEM or
finite-difference methods.

In a stellarator the magnetic field **is not axisymmetric** (∂/∂φ ≠ 0).
The equilibrium condition

```
J × B = ∇p
```

must be satisfied everywhere in 3D. There is no global stream-function ψ, no
simple 2D projection, and the problem is inherently three-dimensional.

### 1.2 Shafranov shift is fully 3D

In a tokamak the Shafranov shift is a scalar `ΔR` (outward displacement of the
magnetic axis as β increases). In a stellarator the magnetic axis position
**R_axis(φ), Z_axis(φ)** varies with the toroidal angle φ. The "shift" is a
full 3D curve that changes shape with β, not just a scalar offset. Ignoring
this makes the 2D approximation progressively worse as β grows.

### 1.3 PerturbGS linearisation is not directly applicable

The `solve_perturbed_gs` / `solve_perturbed_gs_coupled` solvers in
`PerturbGS.py` linearise the MHD equilibrium around a background state
(B₀, J₀, p₀). For this to be self-consistent, **J₀ must satisfy ∇×B₀ = μ₀J₀
and J₀×B₀ = ∇p₀** at every grid point.

In a stellarator:
- B₀ is generated by external coils (no net plasma current) and is fully 3D.
- J₀ is predominantly Pfirsch-Schlüter and diamagnetic current, which varies
  strongly with φ.
- Using only the φ=0 slice of J₀ as if it were axisymmetric introduces O(ε)
  errors per β step (where ε is the helical ripple amplitude, typically 5–20%
  for HAO).

## 2. Current Limitations and Workarounds

### 2.1 2D solve + uniform broadcast

The current implementation in `fenicsx_corrector.py::fpt_fenicsx_beta_step`:

1. Takes a single φ-slice (`phi_slice`, default 0) of the 3D field cache.
2. Solves the 2D linearised force-balance on that slice.
3. Broadcasts the 2D correction `δB(R, Z)` uniformly to **all** φ slices.

This is the **axisymmetric approximation**: it assumes the correction is
the same at every toroidal angle. For a quasi-axisymmetric or near-tokamak
device this may be acceptable at low β; for HAO it introduces systematic error.

### 2.2 Two-slice average for HAO (Np=2)

HAO has field-period Np=2, meaning the magnetic field repeats every π radians.
A better approximation is to:

1. Solve the 2D system at φ=0 → correction `δB₀(R, Z)`.
2. Solve the 2D system at φ=π → correction `δBπ(R, Z)`.
3. Take the average `δB_avg = (δB₀ + δBπ) / 2` as the uniform correction.

This reduces the systematic φ-averaging error by roughly a factor of 2 compared
to the single-slice approach, at the cost of two 2D solves per β step.

**Implementation sketch:**

```python
result_0 = fpt_fenicsx_beta_step(field_cache, beta_old, beta_new, ..., phi_slice=0)
result_pi = fpt_fenicsx_beta_step(field_cache, beta_old, beta_new, ..., phi_slice=n_phi//2)
# Average the corrections
delta_BR   = 0.5 * (result_0['delta_BR'] + result_pi['delta_BR'])
delta_BPhi = 0.5 * (result_0['delta_BPhi'] + result_pi['delta_BPhi'])
delta_BZ   = 0.5 * (result_0['delta_BZ'] + result_pi['delta_BZ'])
```

### 2.3 Correct 3D approach (future work)

The fully correct treatment requires either:

- **3D FEniCSx** mesh covering the full torus (or one field period), solving
  the weak-form MHD equilibrium in 3D. Computationally expensive (~10–100×
  compared to 2D).
- **VMEC** or **HINT2** equilibrium codes: dedicated 3D MHD equilibrium solvers
  that use flux coordinates and are well-validated for stellarators.
- **DESC** (Python, modern): differentiable stellarator equilibrium code that
  can be wrapped for gradient-based optimisation.

Until 3D FEniCSx is available in pyna, the two-slice average is the recommended
operational workaround for HAO.

## 3. Numerical Recommendations

### 3.1 Adaptive `eps_reg`

The regularisation weight `eps_reg` in `solve_force_balance_correction` controls
the Tikhonov regularisation in the FEniCSx linearised solve. Too small → singular
system; too large → over-smoothed, slow convergence.

**Recommended adaptive strategy:**

```python
eps_reg = 1e-6
for attempt in range(4):
    try:
        result = fpt_fenicsx_beta_step(..., eps_reg=eps_reg, ...)
        if result['residual_after'] < 1e-4:
            break
    except Exception:
        pass
    eps_reg *= 10.0  # increase regularisation
    print(f"Newton did not converge, retrying with eps_reg={eps_reg:.0e}")
```

Maximum recommended `eps_reg`: **1e-3** (beyond this the solution is too
regularised to be physically meaningful).

### 3.2 Anderson acceleration depth

Anderson depth m=3 is recommended for HAO:

- The 3D field varies slowly with φ (helical ripple ~10%), so the fixed-point
  iteration converges with modest history depth.
- Larger m (5–10) adds computational cost without proportional benefit.
- Set `anderson_depth=3` in `fpt_fenicsx_beta_step`.

### 3.3 β step size

HAO operates at β ratios around **1% total beta** (β_total ~ 0.01 at full
pressure). Recommended step size:

- **Δβ < 0.5% per step** to keep the linearisation error small.
- At Δβ = 0.5%, the pressure increment is about 50% of the total, which is
  within the linear regime for HAO-scale ripple.
- If Newton diverges (residual grows), halve the step size and retry.

Example β schedule for 0→1%:
```python
beta_steps = np.linspace(0, 0.01, 21)  # 20 steps of 0.05% each
```

### 3.4 Sparse matrix weights in PerturbGS

The BC penalty weights in `solve_perturbed_gs` have been reduced from their
original values to improve condition number:

| Parameter     | Old value | New value | Reason |
|---------------|-----------|-----------|--------|
| `div_weight`  | 1e8       | **1e4**   | κ(A) reduced from >1e20 to ~1e10 |
| `bc_J_weight` | 1e9       | **1e5**   | More balanced with physics rows |
| `bc_p_weight` | 1e12      | **1e6**   | 6 orders of magnitude improvement |

These are now exposed as function parameters and can be tuned per-case.

## 4. Interface with Optuna

### 4.1 Independence of trials

Each Optuna trial corresponds to a different coil current configuration, which
produces a different vacuum field B₀. The β climbing trajectory:

```
β=0 (vacuum) → β=0.5% → β=1.0% → ...
```

is computed on top of this trial-specific vacuum field. Therefore **each trial's
β climb is completely independent** and there is no warm-starting across trials.

### 4.2 Selective β climbing (Pareto filter)

Running the full β climb (20 Newton-corrected FEniCSx steps) for every Optuna
trial is prohibitively expensive. Recommended strategy:

1. Run the main Optuna optimisation loop **without** β climbing (fast, vacuum
   field only), evaluating coil current objectives.
2. Identify the **Pareto-optimal front** (typically ~20 trials out of 200–500).
3. Run the full β climb only on these ~20 Pareto-optimal configurations.

This gives a ~10–25× speedup while still characterising the MHD equilibrium
quality of the best coil designs.

### 4.3 `--with-beta` flag

The `--with-beta` command-line flag (implemented by Agent B in
`hao_optuna_v4.py`) triggers the post-analysis β climb. Usage:

```bash
python hao_optuna_v4.py --with-beta --pareto-top 20 --beta-max 0.01 --beta-steps 20
```

This loads the Optuna study from the database, filters to Pareto-front trials,
and runs `fpt_fenicsx_beta_step` (in WSL2 where dolfinx is available) for each
selected trial.

### 4.4 Output and logging

Each β-climbed trial should record:
- `beta_final`: highest β achieved before Newton divergence
- `residual_after_list`: list of post-Newton residuals per step
- `B_correction_norm_list`: list of correction magnitudes per step

These can be stored as Optuna trial user attributes for later analysis.

## 5. Summary Checklist for HAO β Climbing

- [ ] Use **two-slice average** (φ=0 and φ=π) instead of single-slice broadcast
- [ ] Set `anderson_depth=3` (not 5; saves memory with similar convergence)
- [ ] Set `Δβ ≤ 0.5%/step` (for β_max=1%, use ≥20 steps)
- [ ] Use adaptive `eps_reg`: start 1e-6, retry up to 1e-3 on Newton failure
- [ ] Only run full β climb on Pareto-front (~20) trials, not all trials
- [ ] Prefer GAMG preconditioner (`petsc_solver="gamg"`) in WSL2/Linux FEniCSx
- [ ] Use reduced PerturbGS weights: `div_weight=1e4, bc_J_weight=1e5, bc_p_weight=1e6`
